01 / Executive summary

Build a synchronous ERPNext audit writer, but make the ERPNext object a dedicated DocType.

Recommendation
Known

The webhook seams already exist

Bridge deposit, IBEX crypto receive, and Bridge transfer handlers already perform validation, idempotency, and local persistence/status changes. ENG-348 adds the finance-facing ERPNext write at those seams.

Decision

Use Bridge Transfer Request, not Journal Entry semantics

The ticket asks for a Journal Entry / audit row, but the current need is reconciliation visibility. A dedicated DocType gives Finance a clean report now without prematurely encoding GL policy.

02 / Change map

Three small backend components plus one Frappe DocType.

Frappe DocType

Create Bridge Transfer Request with unique request_id, status/type fields, provider references, Flash references, raw payload, and source-system trace metadata.

ERPNext client + model

Add BridgeTransferRequest model and ErpNext.upsertBridgeTransferRequest with GET-by-request_id, POST, PUT, and duplicate-conflict recovery.

Webhook writer service

Add BridgeTransferRequestWriter mapping functions and call them after existing local persistence/status updates succeed.

04 / Evidence

Code evidence and design evidence used for the recommendation.

GapCurrent webhook handlers stop before ERPNext
Existing handlers have the right webhook points but no ERPNext write yet.

src/services/bridge/webhook-server/routes/transfer.ts

Review note

This is exactly the right insertion point: after local persistence/status updates, before returning webhook success.

DesignDedicated DocType preserves audit semantics without GL coupling
Recommended ERPNext object: Bridge Transfer Request

Core fields:
- request_id unique
- transaction_type: Topup | Cashout
- status: Pending | Fiat Received | Settled | Completed | Failed
- provider: Bridge
- asset: USDT
- network: Ethereum
- amount/currency/developer_fee/receipt amounts
- account_id/wallet_id
- bridge_customer_id/bridge_transfer_id/ibex_tx_hash/address
- source_systems_seen/raw_payload_json/first_seen_at/last_seen_at

Why not Journal Entry in v1:
- ENG-348 needs a reportable audit/reconciliation surface
- GL policy is not finalized
- provider trace payloads fit a purpose-built DocType better than JE remarks/custom fields
- a DocType can later feed formal accounting entries if Finance wants that

Review note

If Finance treats “Journal Entry” as hard acceptance, get explicit sign-off before implementation. Otherwise DocType is the cleaner v1.

05 / Risk review

Main risks are schema sign-off, join keys, and retry observability.

Critical

DocType vs Journal Entry ambiguity

Ticket wording says Journal Entry / audit row. Research recommends a dedicated DocType. Dread/Finance should confirm this distinction before engineering starts.

Warning

IBEX receive may not carry Bridge transfer id

If IBEX crypto receive cannot be joined to a Bridge transfer, v1 should use request_id = ibex: and let ENG-276 link records later.

Note

Network naming changed in practice

Earlier docs mention Tron/TRC-20, but the current PR #344 route validates network=ethereum. ENG-348 should follow the actual ENG-296 implementation branch.
